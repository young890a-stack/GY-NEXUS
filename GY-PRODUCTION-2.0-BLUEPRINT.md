# GY-NEXUS Production 2.0 설계서

## 한 문장 정의

GY는 고객에게 반복해서 쓸 이유가 있는 콘텐츠·AI 도구를 제공하고, Dream Y는 대표 전용 운영실에서 AI 직원·데이터·수익·안전을 지휘하는 Company OS입니다.

## 이번 기반 공사에서 실제로 달라진 것

| 영역 | 이전 | Production 2.0 기반 |
|---|---|---|
| 대표 권한 | 일부 화면만 숨김 | 대표 이메일 검증 + 관리자 API + Service Role의 3중 경계 |
| 데이터 보안 | 과거 SQL에 공개/로그인 사용자 쓰기 정책 존재 | 공개 쓰기 제거, 상품·클릭 집계의 공개 읽기만 허용 |
| AI 회의 | OpenAI 한 모델이 여러 직원 역할을 표현 | Dream Y 전략 생성 후 Gemini가 별도로 위험·수정·보류 판단 |
| 회원 AI | 없음 | Free/Plus/Pro 권한, 월 사용량, 원자적 예약, 실패 기록 |
| 요금제 | 없음 | Free 0원/3회, Plus 9,900원/60회, Pro 29,000원/300회 설계 |
| 고객 신뢰 | 기능 메뉴 중심 | 수익 보장 금지, 사용량 사전 공개, 유료 출시 전 명시적 준비 표시 |

## AI 직원의 진짜 역할

- **Dream Y / OpenAI**: 대표 명령을 실행 가능한 미션으로 분해하고 최종 전략 초안을 만듭니다.
- **Gemini**: 같은 전략을 독립적으로 읽고 사실성, 실행비용, 플랫폼 정책, 고객 가치, 과장 위험을 반대 관점에서 검토합니다.
- **Ethan·Noah·Aurora·Luna·Mia·Atlas·Sophia·Orion**: 현재는 분야별 의견과 미션 소유권을 구조화합니다. 다음 단계에서는 각 직원에게 전용 도구와 성과 기록을 순차 배정합니다.
- 외부 게시, 결제, 대량 실행은 대표 승인 없이 자동 확정하지 않습니다.

## 수입 구조

### 1. 제휴 수익 — 지금 가장 가까운 현금 흐름

상품 페이지 → 추적 링크 → 쿠팡·Temu 등 파트너 전환의 흐름입니다. 클릭 수와 실제 정산 수익을 분리해 기록해야 하며, 클릭을 수익으로 오인하지 않습니다.

### 2. 광고 수익 — 콘텐츠 자산의 누적 효과

Google/Blogger와 Naver에 정보성 콘텐츠를 축적하고 Search Console·GA4로 유입과 체류를 개선합니다. AdSense·AdPost 승인과 수익은 플랫폼 심사 결과에 따릅니다.

### 3. 멤버십 — 반복 방문과 예측 가능한 월 수익

- **GY Free**: 추천, 북마크, 알림, AI 제목 3회
- **GY Plus 9,900원/월**: 제목·개요·쇼츠 기획 60회와 저장·알림
- **GY Pro 29,000원/월**: 콘텐츠·SEO 패키지 300회와 내보내기

가격은 초기 가설입니다. 결제 전환율, AI 원가, 환불률, 재사용률을 확인해 대표가 출시 가격을 확정합니다.

### 4. 후속 수익 — 데이터가 쌓인 뒤 확장

- 브랜드 협찬 캠페인과 검수형 콘텐츠 제작
- 템플릿·프롬프트·운영 자료 같은 디지털 상품
- 소상공인용 GY 운영 워크스페이스와 B2B 월 구독

이 세 가지는 고객 신뢰와 반복 사용 데이터가 확인된 뒤 여는 것이 안전합니다.

## 고객이 다시 오는 반복 구조

1. 검색·추천 상품으로 처음 방문합니다.
2. 무료 회원으로 관심 분야와 북마크를 저장합니다.
3. 매달 갱신되는 AI 체험과 알림 때문에 다시 방문합니다.
4. 실제로 반복 사용하는 고객만 Plus/Pro의 가치를 확인합니다.
5. 문의·사용량·성과 데이터가 다음 추천과 제품 개선에 반영됩니다.

## 통합 상태를 과장하지 않는 기준

| 연결 | 상태 | 의미 |
|---|---|---|
| OpenAI | 실제 사용 | 전략·콘텐츠·회원 AI 생성 |
| Gemini | 이번 버전부터 실제 사용 | 전략 독립 교차검증 |
| Supabase | 실제 사용 | 인증·운영 데이터·회원 권한·사용량 |
| YouTube/Blogger | OAuth 설정 시 실제 게시 | 토큰과 채널 권한이 필요 |
| Search Console/GA4 | 설정 시 실제 조회 | Google 권한과 속성 설정 필요 |
| Runway/OpenAI Image | 키 설정 시 실제 생성 | 생성 비용과 작업 상태 관리 필요 |
| Naver | 부분 연결 | 로그인/게시 패키지 기반이며 공식 블로그 자동 게시를 가장하지 않음 |
| Coupang/Temu | 부분 연결 | 제휴 링크 운영은 가능, 공식 파트너 API는 별도 승인 필요 |

## 다음 공사 순서

1. **결제 준비**: 국내 결제사 선정, 약관·환불·웹훅·영수증·해지 흐름
2. **고객 반복성**: 실제 북마크/조회/알림 통계, 주간 개인화 피드, 이메일 동의 관리
3. **AI 직원 실체화**: 직원별 도구 권한, 입력·출력·비용·성과 로그, 실패 격리
4. **수익 진실화**: 제휴 정산 가져오기, 채널별 순수익, AI 원가 포함 손익
5. **운영 자동화**: 대표 승인 큐, 재시도, 장애 알림, 백업 복구 연습

## 설치 순서

1. Vercel 환경 변수 확인: `SUPABASE_SERVICE_ROLE_KEY`, `OWNER_EMAIL`, `OPENAI_API_KEY`, `GEMINI_API_KEY`
2. 권장 모델 변수 추가: `OPENAI_STRATEGY_MODEL=gpt-5.6-terra`, `OPENAI_MEMBER_MODEL=gpt-5.6-luna`, `GEMINI_REVIEW_MODEL=gemini-3.5-flash`
3. 새 ZIP으로 저장소 파일을 교체하고 GitHub에 커밋·푸시
4. Vercel 재배포가 완료된 뒤 Supabase SQL Editor에서 `supabase/PRODUCTION-2.0-FOUNDATION.sql` 전체 실행
5. 대표 계정과 일반 테스트 계정을 각각 확인

보안 SQL을 새 코드보다 먼저 적용하면 현재 옛 관리자 화면의 데이터 조회가 잠시 막힐 수 있으므로 위 순서를 지킵니다.

## 출시 전 필수 확인

- 일반 회원에게 `/admin`과 관리자 API가 모두 차단되는가
- 일반 회원이 상품 추가·수정·삭제를 직접 할 수 없는가
- Gemini 미연결/실패 시 ‘검증됨’으로 표시하지 않는가
- 월 사용량이 새로고침·동시 클릭에도 초과되지 않는가
- 유료 버튼이 실제 결제 연결 전 돈을 받지 않는가
- 수익 숫자가 클릭·예측·실제 정산으로 구분되는가
